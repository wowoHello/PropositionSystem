# PRD: CWT 命題工作平臺

## 1. Introduction / Overview

**CWT 命題工作平臺**是一個為「全民中檢」設計的線上命題與審題管理系統。平臺提供管理員管理、教師命題、專家審題三大核心工作流程。使用者登入後依據帳號權限顯示對應的功能模組。

本專案目前為**前端 UI DEMO**，所有業務邏輯均以前端模擬方式實現（Mock Data + localStorage），不包含後端 API 與資料庫。預計未來遷移至 **Blazor .NET 10**。

### 目標使用者
- **命題教師**：負責命製試題並進行互審
- **審題委員 (專家學者)**：負責專審階段的題目審查
- **CWT 內部職員 (系統管理員)**：負責專案管理、總審、帳號與權限管理
- 使用者年齡層分佈：壯年至年長者（UI 須適合此年齡層閱讀）

---

## 2. Goals

- 提供完整的命題工作流程：從命題草稿 → 命題完成 → 命題送審 → 三審制度 → 判決結果
- 支援 7 種題型的獨立表單結構（一般、精選、長文、短文題組、閱讀題組、聽力、聽力題組）
- 實現三審制度（互審 → 專審 → 總審）與分歧裁決邏輯
- 提供命題專案（梯次）的生命週期管理
- 提供角色權限控管，依權限顯示對應功能模組
- 提供命題總覽與儀表板以監控命題進度
- 提供系統公告與截止提醒功能
- 提供教師人才庫管理功能
- 確保 UI/UX 適合壯年至年長者使用（字體可調整、配色柔和）

### US-009: 審題任務 - 三審制度
**Description:** 身為一位審題人員，我希望能審閱指派給我的試題，並做出審題決策，以便推進審題流程。

**核心原則：**
- **最終決定權**：所有題目的最終是否採用，皆由「總審」決定
- **按鈕限制**：「互審」階段**不提供**任何決策按鈕（僅給予意見）；「專審」階段**不提供**「不採用」按鈕
- **總審人數彈性**：總審老師原則上為 1 人，但系統需保留彈性，支援未來可能擴充至 2 人以上
- **題目分配迴避**：自己命的題目不會分配給自己審閱；總召若同時為審題人員，不可拿到自己在專審階段審過的題目
- **總審退回容許次數**：總審可退回修改最多兩次，第三次不通過時由總審自行修改並給予「採用」或「不採用」最終標籤

**命審階段時程與鎖定機制：**

| 階段 | 說明 |
|:---|:---|
| 命題階段 | 開放命題教師命題與送審，題目進入待審佇列 |
| 交互審題 | **鎖定**所有命題題目，將待審佇列題目平均分配給梯次命題人員互審（僅給意見） |
| 互審修題 | **解鎖**編輯功能，命題教師依互審意見修改後再次送審 |
| 專家審題 | **鎖定**所有命題題目，將待審佇列題目平均分配給梯次審題人員審閱（可「採用」「改後再審」） |
| 專審修題 | **解鎖**編輯功能，命題教師依專審意見修改後再次送審 |
| 總召審題 | **鎖定**所有命題題目，將待審佇列題目分配給專案梯次負責人與總召人員審閱（可「採用」「改後再審」「不採用」） |
| 總審修題 | **解鎖**編輯功能，命題教師依總審意見修改後再次送審（最多退回兩次，第三次由總審自行修改） |

**審題流程路徑：**

| 流程情境 | 互審 | 專審 | 總審 | 備註 |
|:---|:---|:---|:---|:---|
| 純給意見 | 互審互相提供意見 | - | - | 互審僅給意見，無任何決策按鈕 |
| 互審後需修改 | 給予意見 | 專審 | - | 教師依互審意見修改後送審，進入專審 |
| 專審退回修題 | 通過 | 改後再審 | 總審 | 教師修改後**直接進入總審**（專審僅一次） |
| 專審後需修改 | 通過 | 改後再審 | 總審 | 教師修改後送審，進入總審 |
| 順利通過 | 通過 | 通過 | 總審 | 前兩階段皆通過，進入總審 |
| 總審後需修改（第 1-2 次） | - | - | 改後再審 | 教師修改後再送審，**再次回到總審** |
| 總審後第 3 次不通過 | - | - | 總審自行修改 | 不再退回教師，由總審自行修改並給予「採用」或「不採用」 |
| 總審採用 | - | - | 採用 | 直接進入「審核結果與歷史」，不再開放編輯 |
| 總審不採用 | - | - | 不採用 | 資料移入「審核結果與歷史」 |

**Acceptance Criteria:**
- [ ] 實現三審制度：互審 → 專審 → 總審
- [ ] 互審：命題教師互相審閱，**僅提供意見**（無「採用」「改後再審」「不採用」按鈕），自己命的題目不分配給自己
- [ ] 互審結束後進入專審狀態，已審過的題目按鈕改為「已決策」
- [ ] 專審：專家學者審閱題目，僅提供「採用」與「改後再審」兩種決策（**無不採用按鈕**），自己命的題目不分配給自己
- [ ] 總審：CWT 專員或專案梯次負責人審閱，提供「採用」、「改後再審」、「不採用」三種決策
- [ ] 總審「採用」後，題目直接進入「審核結果與歷史」，不再開放編輯
- [ ] 總審「改後再審」可退回最多兩次；第三次不通過時，由總審自行修改並給予最終決策
- [ ] 總召若同時為審題人員（專審階段），進入總審時不可拿到自己審過的題目
- [ ] 專審退回修題後，教師修改完成再送審時**直接進入總審**（專審僅進行一次）
- [ ] 總審「不採用」後，題目資料移入「審核結果與歷史」供命題老師查看
- [ ] 審題 Modal 內容：題目唯讀（顯示在審題結果之前）、互審/專審/總審意見區塊依階段顯示
- [ ] 支援罐頭訊息快速插入審題意見
- [ ] 點擊 Modal 外部（黑色背景）直接關閉
- [ ] Verify in browser using dev-browser skill

## 4. Functional Requirements

### 登入與導覽
- FR-1: 系統必須提供帳號 + 密碼 + Canvas 驗證碼三因素前端登入驗證
- FR-2: 「忘記密碼」流程：使用者輸入信箱 → 系統寄發含專屬連結的驗證信 → 點擊連結至「變更密碼」頁面；新密碼不可與原密碼相同，違反時以紅字提示
- FR-3: 「使用說明」連結必須以另開視窗方式開啟 PDF 說明文件
- FR-4: 登入成功後跳轉至首頁，並顯示 8 張功能導覽卡片
- FR-5: 所有子頁面共用頂部 Navbar，包含左側 Logo（連結回首頁）、中央專案切換器、右側使用者名稱與登出按鈕

### 命題專案管理
- FR-6: 系統必須支援命題專案（梯次）的新增、編輯、期限管理與人員指派
- FR-7: 專案的「學校」欄位為選填（支援無合作學校的自辦命題專案）
- FR-8: 新增專案時必須展開 8 階段子時程設定（專案區間 → 命題階段 → 交互審題 → 互審修題 → 專家審題 → 專審修題 → 總召審題 → 總審修題），各階段依序連動計算預設天數，且皆可手動調整
- FR-9: 命題階段、互審修題、專審修題、總審修題等 4 個時程階段，在倒數 5 天時需跳出系統提醒
- FR-10: 切換梯次後，所有頁面資料必須依據當前梯次進行篩選與更新

### 命題總覽
- FR-11: 命題總覽頁面僅限「管理者」、「計畫主持人」、「總召老師」、「教務管理者」四種角色可檢視
- FR-12: 「題目內容」區塊必須顯示在「審題結果」之前
- FR-13: 所有原「審題評語」文字統一更名為「審題結果」

### 命題任務
- FR-14: 系統必須支援 7 種題型的獨立表單（一般/精選共用、長文、短文題組、閱讀題組、聽力、聽力題組）
- FR-15: 題目狀態流轉：未儲存 → 命題草稿 → 命題完成 → 命題送審
- FR-16: 點擊 Modal 外部黑色背景時，新增/編輯模式下自動儲存為草稿；檢視模式下僅關閉
- FR-17: 共用 Quill 編輯器以底部滑入方式開啟，所有 `editor-preview-box` 共用同一 Quill 實體
- FR-18: 支援中文標點符號快速插入工具列

### 審題制度
- FR-19: 系統必須實現三審制度：命題階段 → 交互審題 → 互審修題 → 專家審題 → 專審修題 → 總召審題 → 總審修題 → 結案
- FR-20: **互審**（交互審題階段）：鎖定所有命題題目，將待審佇列題目平均分配給梯次命題人員，**僅提供「意見」**（**無採用/改後再審/不採用按鈕**）。自己命的題目不分配給自己
- FR-21: **專審**（專家審題階段）：鎖定所有命題題目，將待審佇列題目平均分配給梯次審題人員，提供「採用」與「改後再審」兩種決策（**無不採用按鈕**）。教師可能同時為命題/審題人員，自己命的題目不分配給自己
- FR-22: **總審**（總召審題階段）：鎖定所有命題題目，將待審佇列題目分配給專案梯次負責人與總召人員，提供「採用」、「改後再審」、「不採用」三種決策。總召可能為複數，且若同時為審題人員，不可拿到自己在專審階段審過的題目
- FR-23: 總審「改後再審」後教師修改再送審，**再次回到總審**（可退回最多兩次）
- FR-24: 總審第三次不通過時，**不再退回教師**，由總審自行修改並給予「採用」或「不採用」最終標籤
- FR-25: 總審「採用」後，題目直接進入「審核結果與歷史」，後續不再開放編輯
- FR-26: 總審老師原則上 1 人，系統需保留彈性支援 2 人以上共同審題
- FR-27: 當教務人員按下梯次「結案」按鈕，僅狀態為「採用」的題目入庫，其餘「改後再審」與「不採用」全部判定為不採用不入題庫
- FR-28: 所有試題進入「審核結果與歷史」Tab 時，題目狀態僅有「採用」、「不採用」二種，且僅可檢視

### 權限管理
- FR-29: 系統必須支援內部人員與外部人員的角色定義與權限設定
- FR-30: 系統必須支援使用者帳號的新增與編輯

### 公告管理
- FR-31: 系統必須支援公告的 CRUD，含分類、置頂、梯次綁定功能
- FR-32: 首頁「今日提醒」必須依當前梯次過濾公告，並將截止倒數項目置於凍結區

### 教師管理
- FR-33: 系統必須支援教師人才庫的維護，包含名單 CRUD、任教背景、命題歷程
- FR-34: 新增的教師必須自動出現在命題專案管理的人員指派選單中

## 5. Non-Goals (Out of Scope)

- 不包含後端 API 與資料庫整合（目前為純前端 DEMO）
- 不包含真實的使用者身份驗證與 Session 管理
- 不包含題庫系統與組卷功能（未來在獨立系統中開發）
- 不包含跨梯次的試題統計分析
- 不包含電子郵件或站內信通知功能
- 不包含試題的自動查重/相似度比對（審題 Modal 內為模擬呈現）
- 不包含多語系支援（僅繁體中文）

## 6. Design Considerations

### 色系
| 用途 | 色碼 |
|------|------|
| 整體背景 | `#F0F2F5`（淺灰色） |
| 導覽列 | `#2563EB`（藍色） |
| 一般裝飾 | 柔和配色，對眼睛友善 |

### UI/UX 原則
- **響應式佈局**：嚴格遵守 Bootstrap 5 Grid System
- **字體**：使用 `Noto Sans TC`，支援字體大小動態調整
- **互動回饋**：所有非同步操作提供 Spinner 或 SweetAlert2 回饋
- **統計卡片**：Hover 時顯示指定顏色的文字變色與外光線動畫
- **點擊外部關閉 Modal**：不使用 `data-bs-backdrop="static"`，點擊背景可關閉

### 現有元件庫
- Bootstrap 5 (CSS + JS)
- Bootstrap Icons
- Quill 富文本編輯器
- SweetAlert2 對話框

## 7. Technical Considerations

### 技術架構
- **Core Stack**: HTML5 / CSS3 / JavaScript (ES6+)
- **UI Framework**: Bootstrap 5 (Utility Classes 優先)
- **Rich Text Editor**: Quill
- **Alert Dialog**: SweetAlert2
- **Data Persistence**: `localStorage`（DEMO 用途）

### 遷移計畫
- 預計未來遷移至 **Blazor .NET 10**
- 程式碼設計保持高內聚、低耦合以利移植
- 避免引入不必要的第三方套件

### 已知限制
- 瀏覽器 CORS 策略會阻擋 `file://` 協定載入 ES6 模組，需透過本地伺服器（`http://localhost:5500`）預覽
- 所有資料均為 Mock Data，實際部署需對接後端服務

### 題型對照表

| 題型 | Handler | 特色 |
|------|---------|------|
| 一般題目 | `GeneralHandler` | 四選一 (ABCD)、主/子類別連動下拉 |
| 精選題目 | `GeneralHandler` | 與一般題目共用 Handler |
| 長文題目 | `LongArticleHandler` | 文章內容 + 試題解析 |
| 短文題組 | `ShortArticleHandler` | 動態新增子題 (手風琴)、主向度/能力指標 |
| 閱讀題組 | `ReadingHandler` | 動態新增子題 (手風琴)、Sidebar 答案同步 |
| 聽力題目 | `ListenHandler` | 難度連動核心能力/指標、四選一 |
| 聽力題組 | `ListenGroupHandler` | 固定兩題子題 (難度三+四)、Sidebar 答案同步 |
